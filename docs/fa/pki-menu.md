# راهنمای Enterprise PKI

این سند تمام گزینه‌های موجود در منوی **PKI Management** را توضیح می‌دهد.

برخلاف پروژه اصلی PD-Element، نسخه **PD-Element Local** دارای یک زیرساخت کامل کلید عمومی (Public Key Infrastructure - PKI) داخلی است که برای استقرار در شبکه‌های محلی سازمانی (Enterprise) طراحی شده است.

این سیستم PKI نیاز به استفاده از Let's Encrypt را برطرف می‌کند و به سازمان‌ها اجازه می‌دهد گواهی‌های مورد اعتماد خود را صادر و مدیریت کنند.

---

# معماری PKI

نصب‌کننده یک ساختار استاندارد Enterprise PKI به‌صورت زیر ایجاد می‌کند:

```
Offline Root CA
│
└── Intermediate CA
│
├── Matrix Certificate
├── Element Certificate
├── TURN Certificate
└── Future Services
```

Root CA باید به‌خوبی محافظت شود و فقط در مواقع ضروری مورد استفاده قرار گیرد.

Intermediate CA مسئول امضای گواهی‌های سرور است.

---

# 1. مقداردهی اولیه PKI (Initialize PKI)

## هدف

ایجاد ساختار کامل PKI.

این عملیات موارد زیر را ایجاد می‌کند:

- دایرکتوری PKI
- Root CA
- Intermediate CA
- پیکربندی OpenSSL
- مجوزهای موردنیاز

## چه زمانی باید از آن استفاده کنم؟

- هنگام اولین نصب
- بازسازی یک PKI از دست‌رفته

## هشدار

روی یک سرور عملیاتی (Production) موجود، یک PKI جدید ایجاد نکنید؛ مگر اینکه قصد داشته باشید تمام گواهی‌ها را جایگزین کنید.

---

# 2. ایجاد Root Certificate Authority

## هدف

ایجاد Root Certificate Authority.

Root CA نقطه اعتماد (Trust Anchor) کل زیرساخت شما است.

## بهترین روش

تا حد امکان Root CA را به‌صورت آفلاین نگهداری کنید.

از Root CA برای امضای مستقیم گواهی‌های سرور استفاده نکنید.

---

# 3. ایجاد Intermediate Certificate Authority

## هدف

ایجاد یک Intermediate CA که توسط Root CA امضا شده باشد.

تمام گواهی‌های سرور باید توسط Intermediate CA امضا شوند.

## چرا؟

نگهداری Root CA به‌صورت آفلاین، امنیت را به‌طور قابل توجهی افزایش می‌دهد.

---

# 4. صدور گواهی سرور (Issue Server Certificate)

## هدف

ایجاد یک گواهی TLS برای سرور.

سرویس‌های پشتیبانی‌شده عبارت‌اند از:

- Matrix Synapse
- Element Web
- Coturn
- سایر سرویس‌های داخلی

نصب‌کننده به‌صورت خودکار از Subject Alternative Names (SANs) پشتیبانی می‌کند.

---

# 5. تمدید گواهی (Renew Certificate)

## هدف

صدور یک گواهی جایگزین پیش از انقضای گواهی فعلی.

بسته به پیکربندی، ممکن است همان کلید خصوصی مجدداً استفاده شود.

## توصیه

برای جلوگیری از اختلال در سرویس‌ها، گواهی‌ها را پیش از تاریخ انقضا تمدید کنید.

---

# 6. بررسی گواهی (Verify Certificate)

## هدف

اعتبارسنجی یک گواهی در برابر زنجیره اعتماد محلی.

موارد زیر بررسی می‌شوند:

- اعتبار امضا (Signature Validation)
- زنجیره اعتماد (Trust Chain)
- تاریخ انقضا
- ورودی‌های SAN
- اطلاعات صادرکننده (Issuer)

---

# 7. ابطال گواهی (Revoke Certificate)

## هدف

علامت‌گذاری یک گواهی به‌عنوان غیرقابل اعتماد.

دلایل متداول عبارت‌اند از:

- افشای کلید خصوصی
- گم‌شدن دستگاه
- جایگزینی سرور

## نکته

پس از ابطال، باید یک گواهی جدید صادر شود.

---

# 8. خروجی گرفتن از Root CA

## هدف

خروجی گرفتن از گواهی Root CA.

این گواهی باید روی سیستم‌های زیر نصب شود:

- Windows
- Linux
- macOS
- Android
- iOS

در صورتی که Root CA مورد اعتماد نباشد، مرورگرها و کلاینت‌های Matrix هشدارهای مربوط به گواهی را نمایش خواهند داد.

---

# 9. خروجی گرفتن از Intermediate CA

## هدف

خروجی گرفتن از گواهی Intermediate CA.

این قابلیت عمدتاً برای موارد زیر مفید است:

- عیب‌یابی
- اعتبارسنجی زنجیره گواهی (Certificate Chain Validation)
- یکپارچه‌سازی با سامانه‌های Enterprise

---

# 10. تهیه نسخه پشتیبان از PKI (Backup PKI)

## هدف

ایجاد یک نسخه پشتیبان امن از موارد زیر:

- Root CA
- Intermediate CA
- کلیدهای خصوصی
- پایگاه داده گواهی‌ها
- CRL
- پیکربندی OpenSSL

## توصیه

نسخه‌های پشتیبان را در چند محل امن نگهداری کنید.

---

# 11. بازیابی PKI (Restore PKI)

## هدف

بازیابی یک نسخه پشتیبان PKI که قبلاً ایجاد شده است.

از این قابلیت پس از موارد زیر استفاده کنید:

- خرابی سرور
- مهاجرت
- بازیابی پس از بحران (Disaster Recovery)

---

# 12. نمایش اطلاعات PKI

## هدف

نمایش اطلاعات مفید مربوط به PKI فعلی.

اطلاعات شامل موارد زیر است:

- Root CA
- Intermediate CA
- مدت اعتبار
- شماره سریال گواهی‌ها
- گواهی‌های صادرشده
- گواهی‌های ابطال‌شده

این اطلاعات برای ممیزی (Auditing) و عیب‌یابی بسیار مفید هستند.

---

# 13. بازگشت به منوی اصلی

از منوی PKI خارج شده و به منوی اصلی برنامه بازگردید.

---

# چرخه عمر گواهی (Certificate Lifecycle)

یک چرخه معمول مدیریت گواهی در محیط‌های Enterprise به شکل زیر است:

1. ایجاد Root CA
2. ایجاد Intermediate CA
3. صدور گواهی سرور
4. نصب گواهی
5. بررسی گواهی
6. تمدید پیش از انقضا
7. ابطال در صورت افشای کلید
8. بایگانی گواهی‌های قدیمی

---

# توصیه‌های امنیتی

- تا حد امکان Root CA را به‌صورت آفلاین نگهداری کنید.
- از تمام کلیدهای خصوصی با استفاده از مجوزهای مناسب فایل محافظت کنید.
- هرگز کلیدهای خصوصی را به اشتراک نگذارید.
- فقط گواهی Root را روی دستگاه‌های کاربران نصب و توزیع کنید.
- پیش از انجام تغییرات مهم، همیشه از PKI نسخه پشتیبان تهیه کنید.
- تاریخ انقضای گواهی‌ها را به‌صورت منظم بررسی کنید.

---

# پرسش‌های متداول

## چرا از Let's Encrypt استفاده نمی‌شود؟

PD-Element Local برای شبکه‌های محلی سازمانی طراحی شده است؛ محیط‌هایی که ممکن است دسترسی به اینترنت یا DNS عمومی در آن‌ها وجود نداشته باشد.

استفاده از یک PKI داخلی، کنترل کامل بر فرآیند صدور گواهی‌ها و مدیریت اعتماد را در اختیار سازمان قرار می‌دهد.

---

## آیا کاربران باید Root CA را نصب کنند؟

بله.

تمام دستگاه‌های کاربران باید به Root CA اعتماد داشته باشند تا هشدارهای TLS نمایش داده نشود و ارتباط امن با سرویس‌های Matrix برقرار شود.

---

## آیا می‌توان PKI ایجادشده را با CA سازمان جایگزین کرد؟

بله.

سازمان‌هایی که از **Microsoft AD CS**، **HashiCorp Vault PKI**، **Smallstep CA** یا هر CA سازمانی دیگری استفاده می‌کنند، می‌توانند PKI ایجادشده را با زیرساخت خود جایگزین کنند؛ مشروط بر اینکه زنجیره گواهی توسط دستگاه‌های کاربران مورد اعتماد باشد.
